# Open Questions (collected during requirements extraction)

- R-ACK-02 (ACK 'only if enabled'): the spec lets a Target ACK or passively NACK its own Dynamic Address. The formal property needs a concrete 'accept_en' signal from the Avalon-MM application layer that gates the ACK driver. Is ACK always asserted on a DA match in v1, or is there an application-controlled accept/busy gate? This must be defined to make the ACK property meaningful.
- R-RD-01/R-RD-04: the read end-of-data condition references the configured MRL (Max Read Length) and IBI payload size set via GET/SETMRL CCCs (Section 4.3.7.3.6), which is outside pages 77-94. Need that section to bound the read-byte counter for a complete formal model.
- R-RD-04: the 150 us read-abandon timeout is a SHOULD and is real-time; expressing it formally requires fixing the system clock frequency and a cycle-count counter. Decide whether v1 implements it and at what tolerance.
- Timing parameters tCAS, tSCO, tDIG_H, tDIG_H_MIXED, tSU_PP (Tables 48/49/50, Section 4.3.11.2) are referenced but not on these pages. The bit-level handoff properties (R-WR-03, R-RD-*, R-IBI-07) abstract these as 'on SCL edge'; confirm whether exact timing must also be checked (likely out of pure-formal scope, into electrical/STA).
- R-WR-02 parity: this section only defines the value the Controller must place on the write T-Bit. It does not explicitly state the Target MUST flag a parity mismatch (that is error-handling / TE type, Section 4.3.8). Confirm whether the endpoint must raise a parity/TE error and how it is surfaced to Avalon-MM.
- Scope confirmation: Group Addressing (4.3.2.1.3) and Virtual Targets / composite devices (4.3.2.1.2) are optional; v1 assumes a single standalone Target with one Dynamic Address. Confirm the matcher needs only {7'h7E, single DA} and no Group Address set.
- Out-of-scope request types: Controller Role Request (DA + RnW=0, requires BCR controller-capable bit) and Hot-Join (7'h02) are excluded in v1. Confirm the endpoint MUST NEVER initiate these (model as: arbitration/IBI engine never drives a CRR pattern or 7'h02), and that this is an assertable safety property.
- Representation of 'passive NACK' and open-drain in the device-agnostic RTL: confirm that NACK = sda_oe=0 (rely on external pull-up = High) and that the Altera tri-state/IO model used in the formal harness represents bus contention safely, so the open-drain ACK and arbitration properties are sound.
- v1 scope: which DA-assignment methods are actually implemented (ENTDAA only, or also SETDASA/SETAASA)? R-DAA-01/R-ADDR-03/R-ADDR-04 hinge on this decision.
- Numeric timing thresholds (tCAS, tBUF, tAVAL, tIDLE, tDIG_L, trDA) for the IBI/Bus-Available/Free counters live in Table 48 and Table 49, which are outside this page range (need the pages with those tables to set concrete cycle counts).
- BCR and DCR are assumed 1 byte each (giving the 8-byte DAA payload shown in Figure 17); confirm exact widths and bit layout from Section 4.3.1.2.2 (outside this range).
- Confirm that 'passively NACK' means releasing SDA to Hi-Z (pulled High) and 'ACK' means driving SDA Low, consistent with the open-drain convention - assumed throughout but defined more fully elsewhere.
- Is Group Addressing in v1 scope? If not supported, R-GRP-02/03/04/05 become N/A, but a defensive decision is needed for how the Target reacts to SETGRPA/group-addressed reads (ignore vs NACK).
- Whether a Target already holding a DA must explicitly NACK 7'h7E in ENTDAA or simply not drive (R-DAA-03) - spec only says undriven devices 'shall ACK'; the non-participation of DA-holding devices is implied, not stated verbatim.
- Restricted Address set (Table 10) needed to enforce R-ADDR-04 (which static addresses forbid SETAASA / cannot be Dynamic Addresses).
- Wake-up requirement R-ACT-02 may be only partially RTL-checkable: 'wake on any Bus activity' depends on clock/power-domain implementation that may sit outside the formally-verified core.
- During ENTDAA, the 7'h7E/R and Repeated START may be Push-Pull (Controller-driven); confirm the Target never needs to push-pull-drive anything during DAA and that its ACK of 7E is always open-drain (assumed in R-DAA-07).
- IBI payload size limit: sec 4.3.6.2.1 cites the Set/Get Max Read Length CCC (4.3.7.3.6) as the bound. It is unclear whether the limit counts the Mandatory Data Byte itself or only the additional bytes after the MDB. Need 4.3.7.3.6 to define the exact count semantics for the R-MDB-04 counter.
- Bus Available / Bus Free / Bus Idle timing thresholds (tAVAL, tIDLE, etc.) are referenced (Table 49, sec 4.3.3.2.3) but defined outside pages 106-117. Exact cycle counts are needed to parameterize the IBI START-request and retry timers (R-IBI-04, R-IBI-05).
- Whether a v1 SDR Target must support a Mandatory Data Byte at all is a design choice encoded in BCR[2]: spec makes MDB conditional on BCR[2]. Need to decide BCR[2] value; this changes which of R-MDB-01 vs R-MDB-02 applies and whether the push-pull payload datapath exists.
- Sec 4.3.6.2 says the Target arbitrates its address 'following a START'. Sec 4.3.6.4 shows the controller opening arbitration with 7'h7E. It is ambiguous whether the Target may arbitrate an IBI after a plain START in which the controller drives a specific (non-7E) target address (i.e., a directed read/write), or only when the header is 7'h7E. This affects when R-TOL-01 / R-IBI-01 permit driving the address.
- Arbitration tie-break behavior (controller drives identical address/RnW, no clear winner) is deferred to sec 4.3.2.2.3 (outside this range). Need that section to define exactly how the Target detects the inconclusive case and re-arms (R-TOL-03).
- Pending Read Notification enable/disable mechanism is explicitly 'not defined by the I3C Basic Specification' (private contract or MDB assignment). No formal property can pin the enable gating beyond ENINT; the prn_send_enabled control is implementation-defined (R-PRN-09).
- Exact ACK/NACK sampling point and SDA-vs-SCL setup/hold for the IBI address+RnW and the push-pull MDB are governed by sec 4.3.2.x timing (outside this range); needed to make the ACK-detection and push-pull-handoff properties (R-IBI-06, R-MDB-01) cycle-accurate.
- T-bit definition for IBI data references sec 4.3.2.3.4 (outside range): confirm whether the MDB itself carries a T bit and its parity/odd-even semantics so R-MDB-03 is encoded correctly.
- Exact 'in time' threshold for the Direct GET single-retry (R-GET-NACK-01): the spec on p.88 leaves the response-ready deadline implementation-defined; the formal model needs a chosen latency/state boundary.
- Broadcast CCC 'ignore' vs 'unsupported': there is a single 7'h7E ACK for the whole broadcast (all Targets ACK), so a Target cannot per-address NACK an unsupported Broadcast CCC. How a Target signals/handles an unsupported BROADCAST CCC (vs the explicit Direct-CCC NACK rule) is not defined in this range - need 4.3.7.3 / Table 16.
- Strength of R-DEFB-ZERO-01: the text says 'shall generally treat a Defining Byte value of zero as though the Defining Byte were not present' - 'generally' plus the carve-out for Required-Defining-Byte CCCs makes it ambiguous whether this is MUST or SHOULD; modeled here as SHOULD.
- Whether Group Address recognition (R-ADDR-RECOG-01(c), R-UNSUP-CLASS-01 group case) is in scope for v1: spec marks Group support optional; project scope (SDR+IBI, no secondary controller/Hot-Join) likely omits Group addressing - confirm before asserting group-match properties.
- Relationship between unsupported-CCC NACK (4.3.7.2.2) and error type TE5 illegal-CCC handling (4.3.8.1.6, outside range): spec notes they can look identical to the Controller but are not the same; the precise Target-internal distinction must be resolved from Section 4.3.8.
- Exact T-Bit semantics for Direct Read/GET (R-TBIT-READ-01, R-DIR-READ-RESP-01) depend on SDR Read model details in Section 4.3.2.3.4 (outside range) - parity vs early-termination handoff must be pulled from there to write a precise SVA property.
- Premature/invalid CCC termination (R-CCC-PREMATURE-01): 'best efforts' and the disregard-vs-process choice are explicitly implementation-defined, so only a no-hang/no-contention safety property is assertable; the data-processing policy is a design decision.
- Sub-Command Byte applicability (R-SUBCMD-01): which, if any, of the supported v1 Direct Write/SET CCCs actually use Sub-Command Bytes is defined per-CCC in Section 4.3.7.3 (outside range); needed to decide whether the parser must implement them.
- GETMXDS (4.3.7.3.18) and GETCAPS (4.3.7.3.19) detailed return formats are PAST page 155 (printed 119); only Table 16 conditions are visible here. Part B (pages ~156+) must be read to extract: exact GETCAPS byte/bit map (esp. GETCAP3[4]=GETSTATUS-Format2 support, CRCAPS bits), GETMXDS data format (max RD/WR speed codes, optional Read-Turnaround byte), and which capability bits a v1.2 Target MUST set.
- ENEC/DISEC Events Byte bit positions render in the PDF table as 'Reserved ENHJ Reserved ENCR ENINT' across Bit7..Bit0; this matches full-I3C convention ENINT=bit0, ENCR=bit1, ENHJ=bit3 (bit2 and bits[7:4] Reserved), but the exact reserved-bit placement should be confirmed against Table 18/19 graphics. Confirm the Target must silently ignore set reserved/unsupported event bits (no protocol error).
- Reset/default values for MWL and MRL registers are not numerically specified here — spec only says the Target 'should define the minimum possible value' (SHOULD). Needs a vendor/product decision (default MWL, default MRL, min/max bounds, and whether out-of-range writes are clamped or simply ignored).
- Whether an out-of-range SETMWL/SETMRL value should be NACKed vs silently left-unchanged is only a SHOULD ('leave unchanged'); the spec does not require NACK. Confirm desired implementation policy.
- RSTACT (Required per Table 16) and the Target Reset Pattern detection are detailed in §4.3.9 (outside 125-155); the surrounding stale extract showed §4.3.9.3/.4 reset-scope tables (Table 45) — must be formally read in the Part B assignment to extract exact reset scopes (0x01 peripheral vs 0x02 whole-target) and Target Reset Pattern (14 SDA transitions while SCL low) requirements.
- GETSTATUS LSB[7:6] Activity Mode meaning for ordinary (non-secondary-controller) Targets is explicitly 'a private contract between Controller and each Target' — undefined by the spec; the endpoint product must define these 2 bits (or tie to 0).
- The Retry Model for Direct GET CCCs (§4.3.7.2.3) referenced by GETSTATUS Format 2 (NACK-first-then-single-retry) is outside this page range; exact retry counting/timing must be confirmed from §4.3.7.2.
- General rule for unsupported Direct CCCs (NACK own Target Address to terminate) is inferred from per-CCC text and figures; the explicit normative statement of 'NACK address => terminate Direct CCC for this Target' lives in §4.3.7.2 (CCC framing), which should be cross-checked for the global ACK/NACK obligation.
- BCR bit meanings used in several conditions (bit0=max-data-speed-limit -> GETMXDS required; bit2=IBI-payload -> GETMRL 3rd byte) are defined in §4.3.1.2.1 (outside this range); confirm the exact BCR bit map to wire the conditional CCC support correctly.
- GETCAPS command code is not printed within pp.156-185 (the section says it replaces GETHDRCAP and is detailed in Section 5.4). Confirm the code (commonly 0x95) and confirm whether a minimal SDR+IBI Target must implement any NACK-on-unsupported-Defining-Byte behavior for GETCAPS Format 2 analogous to GETMXDS.
- GETMXDS for a BCR[0]=0 Target: the spec says support is 'required only when BCR[0]=1', implying optional otherwise, but does not explicitly state the Target must NACK if addressed when unsupported. We assumed NACK via the general Direct-CCC model (4.3.7.2.3). Confirm against Section 4.3.7.2.2/4.3.7.2.3 (outside this range).
- RSTACT Direct-Read (Format 3) return-time support: is it ever mandatory for a v1.2 Target, or always optional? p.140 phrases it as conditional ('If the Target supports the Direct Read format...'). Confirm that NACKing 0x81-0x83 is fully compliant for a minimal Target.
- ENDXFER is marked 'Conditional' (Table 35) but the precise condition for a SDR-only/no-HDR/non-Monitoring Target is not spelled out here; all listed Defining Bytes (0x7F,0x55,0xF7,0xAA,0xFC) are HDR/monitoring/MIPI-private. Confirm there is no SDR-mode obligation for ENDXFER in Sections 6.2.3.4 / 6.x before treating it as fully NACK-able.
- Broadcast CCC ACK semantics: for SETAASA/SETBUSCON/RSTACT-Broadcast 'ignore' we assumed no per-Target NACK (only 0x7E is ACKed). The exact broadcast ACK/T-bit framing is defined in 4.3.7.x framing sections outside pp.156-185 — confirm the bit-level expectation so the FSM does not mistakenly drive ACK/NACK on broadcast data bytes.
- Group-address Direct-Write ACK position/timing references 'the same manner as a Direct CCC Write to its Dynamic Address (per Section 4.3.7.2.2)', which is outside this range. Need that section to pin the exact ACK bit cycle for the formal property R-GRP-02.
- START vs Repeated START discrimination for R-RSTACT-03: the requirement is bus-level (true START clears the pending reset action). Confirm the Target's S/Sr detector definition (Section 5.1.x bus condition timing) so the formal property's start_detected/repeated_start_detected signals are well-defined.
- Whether 'react only to recognized context' (SETBUSCON) imposes any mandatory action for the I3C-Basic-version context values (Table 41, 1-63) on a v1.2 Target, or whether ignoring all context values is compliant for a minimal Target.
- TE2/TE5/TE6 post-CCC recovery each offer TWO allowed behaviors (retain CCC state until Repeated START vs. discard until STOP). Which option does v1 implement? It must be fixed before formal properties can be written precisely (R-TE2-02, R-TE5-02, R-TE6-05).
- TE4 wording conflict: body (4.3.8.1.5, p.152) says NACK then 'wait for STOP to exit ENTDAA', but Table 43 (p.150) says 'enable STOP or Repeated START Detector and ignore all other patterns'. Does the Target also accept a Repeated START as a TE4 recovery, or only STOP?
- Whole-chip escalation tension: 4.3.9.1 (p.161) says on the 2nd Target Reset Pattern the Target 'shall reset the whole chip', but Table 45 note 3 (p.164) says Whole Target reset support is OPTIONAL. If v1 does not support whole-target reset, what is the conformant default-escalation behavior (stay at peripheral reset? NACK?)?
- Reset-state persistence: the escalation 'armed' flag (R-TRST-ESC2) and the reset-action register must survive a Peripheral reset. Which reset domain / always-on logic holds these? This drives the RTL reset partitioning and matters for the formal model's reset modeling.
- Analog timing thresholds (60 us both-lines-High for TE0/TE1 recovery; 150 us SCL-hold read-abort; 50 ms tCAS) are not naturally expressible in cycle-accurate SVA. Decide whether to model them as parameterized cycle counters against a known clock, or exclude from formal and cover by simulation. All three are optional/secondary-controller anyway.
- Peripheral-reset memory retention is explicitly implementation-defined (spec silent, Table 45). Need a product decision on whether the Dynamic Address (and which config registers) are retained vs cleared on a Peripheral reset — this changes R-TRST-PERIPH-DA and the ENTDAA-participation property.
- RSTACT Defining Byte values 0x03 (Debug Network Adaptor Reset) and 0x04 (Virtual Target Detect): does the v1 endpoint need to act on these, or treat them as unsupported (ignore / no-op / NACK)? Same question for vendor range 0x40-0x7F.
- Is RSTACT support (all three forms incl. Directed Read returning reset-recovery timing) mandatory for a minimal v1 Target, or can the Target rely solely on default escalation? Need Section 4.3.7.3.23 and the Required-CCC table (outside this page range) to size the GET-timing response.
- TE6 monitoring (R-TE6-01/02) requires sensing the actual SDA line level vs the intended driven bit. Does the FPGA I/O provide reliable readback of the open-drain line for the formal/RTL model, and is TE6 in v1 scope at all (it is optional)?
- HDR Exit Pattern detector is mandatory for TE0/TE1 recovery and is the basis of the Target Reset Pattern, but its exact definition (number of SDA transitions, =4?) is in Section 4.3.10.2 (outside pp.186-202). Need that section to correctly size the shared SDA-transition counter and the 14-vs-shorter distinction (R-TRP-DISTINGUISH).
- Accurate START vs Repeated START discrimination is load-bearing for R-TRST-CLRCFG (clear configured reset action on START's SCL-fall but NOT on Sr) and for R-TE0-01 header sampling. Confirm the Target's START/Sr detector resolution is sufficient and how it is modeled formally.
- The set of 'Target Error states recoverable by the HDR Exit Pattern' (R-HDREXIT-04 case B) is defined in Sec 4.3.8.1, which is outside this page range (203-232); the exit-detector enable logic must cross-reference that section.
- Spec says enable the exit detector on 'any ENTHDRx CCC' - confirm the Target only needs to detect that an ENTHDRx (0x20-0x27) occurred, not decode which specific HDR mode, since v1 supports none.
- tDIG_H min = 32 ns sets the minimum inter-edge spacing of the HDR Exit Pattern, but the spec does not prescribe a sampling/oversampling clock rate for a synchronous FPGA detector. Need to fix an internal sample clock (e.g. >= ~50-100 MHz) and prove the exit/START/STOP detectors meet it. Open: required oversampling factor and metastability sync depth.
- The reference detector (Listing 1) uses SDA as a clock and an async reset on SCL. For a clean open-source formal flow (yosys/SymbiYosys) and Altera STA, should this be re-implemented as a fully-synchronous oversampled FSM? If so, R-HDREXIT-02/07 properties must be restated over sampled signals rather than edge-clocked flops.
- Spike filter applicability: Table 47 marks the 50 ns filter 'Desirable' (not Required) for legacy I2C, and Fig 90 shows I3C push-pull edges are deliberately shorter than 50 ns. For an SDR-only I3C target with no I2C support, is any input glitch filter expected, and if so must it be disabled before high-speed SDR so it doesn't erase valid 24-32 ns half-periods?
- tAVAL/tIDLE/tBUF behavioral usage (when exactly the Target may drive an IBI START) is only given as values here; the gating sequence is defined in the IBI sections. The IBI-START timing property (R-BUS-01) needs the IBI section to be pinned down before writing the full SVA.
- tSCO > 20 ns path (R-TIM-02): need to confirm whether the v1 FPGA implementation can guarantee tSCO <= 20 ns; if not, BCR limitation bit + GETMXDS/maxRD support becomes mandatory and must be added to the register map and CCC handler scope.
- Push-pull vs open-drain drive-mode selection per frame phase (R-DRV-01/R-ARB-01) depends on whether the Controller chose push-pull for Addr[5:0]/RnW (no A6 contention) - the Target only observes these phases as inputs except during ACK/read-data, so confirm the Target never needs to drive during the arbitrated address bits other than its own IBI arbitration.
- GETCAP3 Format-2 indicator bit ambiguity: prose on p.206 (phys 242) says a Format-2-capable Target 'shall return 1'b1 in bit 4 of Byte 3 (GETCAP3)', but Table 63 (p.205) labels bit 3 as 'Defining Byte Support in GETCAPS' and bit 4 as 'Defining Byte Support in GETSTATUS'. Which physical bit must encode GETCAPS-Format-2 support? (R-GCAP3-DBGETCAPS assumes bit 3 per the table; resolve before RTL freeze.)
- Will v1 support GETCAPS Format 2 at all? If only Format 1 (TGTCAPS) is implemented, define exact behavior when the Controller sends a Format-2 Defining Byte: does the Target NACK the address, or treat any DB!=0x00 as unsupported -> NACK? The spec only mandates NACK for specific unsupported feature DBs (CRCAPS/VTCAPS/DBGCAPS); reserved-DB behavior for a Format-2-incapable Target is not spelled out in this section.
- Is TESTPAT (DB 0x5A) supported in v1? It is only Recommended (for devices on a bus with Debug-capable devices), not required. If unsupported the Target NACKs DB 0x5A; confirm product decision.
- DCR constant: should the generic endpoint ship with DCR=0x00 (Generic Device) or a registered device-type code? The actual numeric constant depends on MIPI DCR registry assignment for the product.
- IBI MDB constant(s): which Interrupt Group and Specific Interrupt Identifier will this endpoint emit (vendor 3'b000, I3C-WG 3'b001 error codes, or pending-read 3'b101)? This drives BCR[2], GETCAP3[6], and the MDB generator constants.
- RSTACT GET return-time constants (DB 0x81 'Return Time to Reset Peripheral', 0x82 'Return Time to Reset Whole Target') and the actual reset action semantics are defined in Section 4.3.7.3.x (outside pages 233-254). Need those values/encoding to complete the RSTACT response and reset FSM.
- Timing Control: confirm the endpoint does NOT support Timing Control, so GETCAP3[7]=0 and SETXTIME/GETXTIME are NACKed. (Affects R-GCAP3-TC.)
- Optional Defining-Byte support for GETSTATUS (GETCAP3[4]) and the optional 'Maximum Read Turnaround' field of GETMXDS DB 0x00 - implement or not in v1?
- SETBUSCON: is the Target required to take any action on the received context byte in SDR scope, or merely tolerate it? Treated here as tolerate-only (R-BUSCON-01); confirm no functional dependence.
- ENTTM Vendor Test Mode (DB 0x01) mandates returning a RANDOM 32-bit value in the Provisioned ID during DAA - if Test Mode is supported, a random source is needed and this behavior is not formally verifiable. Confirm Test Mode is out of v1 scope.
- Read-data T-bit polarity vs. early-abort: F.1 (Fig 189) states T=1 means 'additional data' and T=0 'signals the end', and the Target releases SDA after T. Need the Section 5/6 SDR read framing to confirm exactly when the Target must drive T=0 (only on its genuine last byte) vs. how it interacts with the Controller's abort (pull SDA Low = Sr). Annex F alone is ambiguous on whether the Target ever forces termination via T=0 mid-stream.
- ENTDAA ACK 'with Handoff' vs 'without Handoff' on the assigned address (Figs 175/180 legends): confirm that a Basic (non-secondary-controller) Target always ACKs without Handoff and what bit/encoding distinguishes the two, since v1 must guarantee it is never handed the controller role.
- Exact byte order and field widths of the ENTDAA 8-byte payload: Annex C calls it '48-bit Unique ID, BCR, DCR' (Fig 175) read as 8 bytes. Confirm whether it is PID[47:0] (6 bytes) + BCR + DCR and the MSB-first ordering at bit granularity for the arbitration comparator (Section 5.1.4 needed for exact bit map).
- TE0 restricted-address set is defined for a Target that has a dynamic address assigned; need to confirm Target behavior BEFORE any dynamic address is assigned (pre-DAA): the spec says TE0 detector is used 'after a Dynamic Address has been assigned'. What address-decode/ignore policy applies to a freshly powered, unassigned Target outside ENTDAA?
- HDR Exit Pattern detector is required for SDR-only Targets to recover from TE0/TE1/HDR entry (R-ERR-TE0/TE1, R-TOL-HDR), but the exact HDR Exit Pattern definition is in Section 5.2/6 (outside pages 467-492). Need that definition to implement the recovery detector and to write a concrete SVA for the exit.
- IBI MDB content/encoding and whether additional data bytes use the same End-of-Data T-bit semantics as Private Read: Fig 182 shows 'Read MDB and any add'l Data Bytes' but the MDB format and the optional-byte termination rules are defined in the IBI section of the main body (Section 5.1.6), not in this Annex range.
- Directed-CCC direction-legality table (for TE5): the Annex only gives examples (GETBCR is read-only, SETGRPA is write-only). The full per-CCC SET/GET/Read/Write legality matrix (Section 5.1.9.3 CCC tables) is required to implement the TE5 illegal-format detector precisely.
- Whether v1 must implement the optional >60us SCL+SDA-High recovery (R-ERR-RECOVERY-60US) and TE6 read-back monitoring (R-ERR-TE6); these are SHOULD/MAY and may be deferred, but if implemented they add timers/comparators that should be in scope decisions before RTL.
